# delego Wire Specification — protocol 0.1

**Version:** 0.1 · **Status:** Superseded by [v0.2](spec-v0.2.md) and [v0.3](../spec.md) · **License:** Apache-2.0

> **About this document.** This is the specification of **protocol 0.1**, the
> initial protocol version. It was **backfilled retroactively** from the frozen
> v0.3 document by removing every clause tagged *(since 0.2)* or *(0.3, draft)*,
> so that each protocol version has a standalone spec of record; it is not the
> historical 0.1-draft text (that text predates the genericized examples and
> carried a since-corrected §8.1 truncation claim — see the repository's git
> history for the original). Section numbering matches [`spec.md`](../spec.md);
> sections that exist only in later versions are stubbed. The current
> specification is **[spec.md (v0.3)](../spec.md)**.
>
> ⚠️ **Known 0.1 limitations, closed in 0.2** (§7): a 0.1 approval is bound to
> the action fingerprint but **not** to the `intent_hash`, and is **not
> single-use** — an `approved` approval can release its action repeatedly.
> Implementations SHOULD move to 0.2.

This document specifies the delego protocol for **intent-bound action
authorization**: every action an autonomous agent proposes is authorized
deterministically — with no language model in the decision path — and before any
credential is used, with the authorization cryptographically bound to the
originating human instruction and the exact action, and recorded in a
tamper-evident, signed audit chain. delego is a deterministic pre-action
authorization layer for AI agents; it sits between an agent that proposes actions
and the credential broker that executes them, and answers, for each action,
`allow` / `needs_approval` / `deny` as a pure function of (action, policy, audit
ledger, time), bound to the action's fingerprint so it cannot be transferred to a
different action. This addresses the agent-authorization gap catalogued as OWASP
ASI03 (Excessive Agency).

The goal is **interoperability**: an authorizer, a credential broker, and an
auditor written by different parties, in different languages, must agree
byte-for-byte. The companion [Conformance Test Kit](../ctk/README.md) provides
authoritative test vectors generated by the reference implementation
([`delego`](https://github.com/Delego-Dev/delego)).

## 1. Conventions

delego is a **deterministic pre-action authorization layer for AI agents**: it
decides `allow` / `needs_approval` / `deny` for each proposed action, with no
language model in the decision path, before any credential is used.

The key words **MUST**, **MUST NOT**, **REQUIRED**, **SHALL**, **SHOULD**,
**SHOULD NOT**, **MAY**, and **OPTIONAL** are to be interpreted as described in
[RFC 2119](https://www.rfc-editor.org/rfc/rfc2119) and
[RFC 8174](https://www.rfc-editor.org/rfc/rfc8174).

## 2. Roles & trust model

| Role | Responsibility |
|------|----------------|
| **Agent** | Proposes actions. Untrusted: may be steered by a prompt injection. |
| **Authorizer** | Evaluates a proposed action against a policy and renders a deterministic decision; parks sensitive actions for human approval; writes the audit chain. (Reference: `delego`.) This is the **Policy Decision Point (PDP)**. |
| **Human approver** | Out-of-band, releases or refuses a parked action through a channel the Agent does not control. |
| **Broker** | Holds the credential, injects it, and forwards the request upstream. This is the **Policy Enforcement Point (PEP)**. The Authorizer never holds credentials. |
| **Service** | The upstream API the action targets. |
| **Auditor** | Verifies the audit chain after the fact. |

**PDP/PEP split.** delego separates the *decision* from the *enforcement*. The
**Authorizer is the Policy Decision Point (PDP)**: it renders the deterministic
verdict and never holds a credential. The **Broker is the Policy Enforcement
Point (PEP)**: it is the *only* component that touches a credential, and it
injects one only after the PDP has authorized that exact action. Because the
credential is unreachable except through the PEP, and the PEP acts only on a PDP
verdict bound to the action's fingerprint (§4), an Agent that is redirected
cannot reach a credential on its own authority.

**Trust boundary.** The decision that gates a credential is made by the
Authorizer in deterministic code. An Authorizer **MUST NOT** consult a language
model, or any non-deterministic input, when rendering a decision (§6). The Agent
is assumed compromisable; the protocol's job is to bound blast radius and prove
provenance, not to infer the human's intent semantically.

## 2.1 Protocol versions

delego is versioned so the **specification can lead the reference
implementation**. A behaviour is specified *first* and becomes reference-backed
only once the implementation reproduces its Conformance Test Kit vectors (§10).
The reference's protocol version (`delego.__protocol_version__`) therefore
**MUST** always be ≤ the version of the specification.

| Version | Status | Adds |

## 3. Canonicalization (NORMATIVE)

Every value that is hashed or signed is first serialized with **canonical JSON**:

- UTF-8 encoded;
- object keys sorted lexicographically by Unicode code point;
- no insignificant whitespace — item separator `","`, key/value separator `":"`;
- non-ASCII characters are **not** escaped (equivalent to `ensure_ascii=false`).

A hash is the lowercase, base-16 **SHA-256** digest of the UTF-8 bytes of a
canonical-JSON string. This document writes it `sha256(canonical_json(x))`.

Two implementations serializing the same logical value **MUST** produce
identical bytes; otherwise hashes and signatures will not agree.

## 4. Proposed Action

A proposed action is an object:

| Field | Type | Notes |
|-------|------|-------|
| `instruction` | string | The original human instruction the action serves. |
| `method` | string | HTTP method. |
| `url` | string | Full target URL. |
| `params` | object | Decision-relevant request fields (e.g. `amount`, `currency`). |

From it, two values are derived.

**Intent hash** — binds the action to the human's words:

```
intent_hash = sha256(canonical_json({ "instruction": TRIM(instruction) }))
```

`TRIM` removes leading/trailing whitespace.

**Action fingerprint** — the exact identity of the action; what approvals are
bound to:

```
host = lowercase(authority component of url)
path = path component of url, or "/" if empty
action_fingerprint = sha256(canonical_json({
  "method": uppercase(method),
  "host":   host,
  "path":   path,
  "params": params
}))
```

### 4.1 Worked example (authoritative)

For `GET https://api.example.com/accounts/me`, instruction `"read my account details"`,
`params = {}`:

```
canonical_json for fingerprint:
{"host":"api.example.com","method":"GET","params":{},"path":"/accounts/me"}

intent_hash        = ec949034e985a92f3bcd9f9ab8313a80005157698f748f2b8df6163c04af4619
action_fingerprint = 497e02606ea157ca8ca885cbbd33d1a8a70c40fdaf6f15c5154d858f870b8b61
```

Adding one parameter changes the fingerprint completely — this is what makes the
confused-deputy guard (§7) work:

```
order {amount:2400,currency:USD,destination:internal}
  → c70d4ee57957202087887cb5e9d32222977b728bd06947b7761c283b6d4ed394
order {…, recipient:"attacker"}    (one param added)
  → dabddc8fc7e8fb30bdec6fb796a336b7897d4a2a12ae386727e2110d7e0e9572
```

See [`ctk/vectors/hashing.json`](../ctk/vectors/hashing.json) for the full set.

### 4.2 Query string

In 0.1 the fingerprint covers `method`, `host`, `path`, and the agent-declared
`params`; the URL's **query string is not part of the fingerprint**. Two
requests that differ only in their query (e.g. `/orders?to=me` vs
`/orders?to=attacker`) therefore share one fingerprint — a confused-deputy gap
if decision-relevant data rides the query.

Decision-relevant values **MUST** therefore be carried in `params`, not solely
in the query, so they are fingerprinted and policy-evaluated.

> **Hardened in v0.3.** The normative Broker query obligation and the eventual
> folding of the query into the fingerprint preimage are specified in
> [spec.md §4.2](../spec.md#42-query-string).

## 5. Policy

A policy is a document (YAML or JSON) conforming to
[`schema/policy.json`](../schema/policy.json):

```yaml
version: 1
default: deny            # outcome when nothing matches; SHOULD be "deny"
forbidden:               # hard denials, evaluated first
  - name: no-access-control-changes
    match: { path_contains: /permissions }
rules:                   # first match wins
  - name: place-order
    decision: needs_approval
    match: { method: POST, host: api.example.com, path: /orders }
    constraints:
      amount:     { field: amount, max: 5000, currency: USD }
      allow_list: { field: destination, in: [internal] }
```

A **rule** has a `name`, a `decision` (`allow` or `needs_approval`), a `match`,
and optional `constraints`. A **match** MAY constrain `method` (string or list),
`host` (exact, case-insensitive), `path` (glob), and `path_contains` (substring);
an empty match matches nothing.

**Constraints** (all fail-closed):
- `amount` — `{ field, max, currency }`. The named field MUST be present and
  numeric; if `currency` is given, the action's `currency` MUST equal it; the
  value MUST be ≤ `max`.
- `allow_list` — `{ field, in: [...] }`. The named field's value MUST be a member.
- `rate_limit` — `{ max, per: minute|hour|day }`. Counts prior `allow` receipts
  for the same rule within the window (§8); denies at or above `max`.

> **Glob note (v0.1).** Path globbing is coarse: `**` and `*` are treated alike
> and both span `/`. Per-segment globbing is a planned refinement; a conformant
> v0.1 implementation MUST reproduce the coarse behavior.

> **Hardened in 0.2 / v0.3.** 0.2 requires an unevaluable `rate_limit` (no audit
> ledger available) to `deny` rather than silently pass, and counts
> approval-released executions against the cap ([spec-v0.2.md §5](spec-v0.2.md#5-policy)).
> v0.3 adds normative load-time policy validation and the `rate_limit`
> consistency-class requirement ([spec.md §5.1, §5](../spec.md#51-policy-validation-normative-03-draft--additive)).

## 6. Decision (NORMATIVE)

Evaluation order is fixed:

1. **`forbidden`** — if any forbidden entry matches, the outcome is `deny`.
2. **`rules`** — the first rule whose `match` matches is selected. Its
   constraints are checked; if **any** fail, the outcome is `deny`
   (fail-closed). Otherwise the outcome is the rule's `decision`.
3. **`default`** — if nothing matched, the outcome is `default`.

The result is `{ outcome, rule, reasons }` where `outcome` ∈ `{allow, deny,
needs_approval}`, `rule` is the matched rule name (or null), and `reasons` is a
list of human-readable strings. Evaluation **MUST** be a pure, deterministic
function of (action, policy, audit ledger, evaluation time) — there is no
stochastic or hidden input. (The `rate_limit` constraint is the only one that
reads the evaluation time, to size its window; all others depend on action and
policy alone.)

Authoritative decision vectors: [`ctk/vectors/decisions.json`](../ctk/vectors/decisions.json).

## 7. Approval binding (the confused-deputy guard)

When the outcome is `needs_approval`, the Authorizer parks the action — bound to
its exact `action_fingerprint` — and returns an `approval_id`. A human decides
out-of-band. An approval moves through a status lifecycle:

```
pending ──approve──▶ approved
   └─────deny──────▶ denied
```

When the Agent later presents an action to *release* an approval, the Authorizer
**MUST** evaluate:

1. **Fingerprint guard.** Recompute the presented action's `action_fingerprint`
   and compare it to the one the approval was issued for. **If they differ, the
   outcome MUST be `deny`** — the approval does not transfer to a different
   action. This check is made **before** the approval's status is considered: a
   substituted action is refused whether or not a human has approved the original.

2. **Status.** Then resolve by the approval's status:
   - `pending` → outcome `needs_approval` (no human decision yet);
   - `denied` → outcome `deny`;
   - `approved` → release the action and emit an `execution` receipt (§8).

Every refusal in this section **MUST** be recorded as an `execution`/`deny`
receipt (§8).

> ⚠️ **Closed in 0.2.** 0.1 approvals are **not single-use** — an `approved`
> approval has no `consumed` state, so a release can be replayed — and are
> **not bound to the `intent_hash`**, so the same action can be re-presented
> under a different claimed instruction. 0.2 adds the intent guard, the
> `consumed` terminal status, and rule attribution on the released receipt
> ([spec-v0.2.md §7](spec-v0.2.md#7-approval-binding-the-confused-deputy-guard)).

## 8. Receipt & audit chain (NORMATIVE)

Every decision and execution is recorded as a **receipt**. Receipts form an
append-only, hash-linked, signed chain conforming to
[`schema/receipt.json`](../schema/receipt.json).

The **signed payload** is exactly these eleven fields:

```
seq, ts, phase, outcome, rule, reasons,
intent_hash, action_fingerprint, action_summary, approval_id, prev_hash
```

- `seq` — 0-based, contiguous, monotonically increasing.
- `phase` — `decision` or `execution`.
- `prev_hash` — the previous receipt's `entry_hash`, or the literal `"GENESIS"`
  for `seq` 0.

Two derived fields complete a receipt:

```
entry_hash = sha256(canonical_json(payload))           # payload = the 11 fields
signature  = hex( Ed25519_sign( privkey, UTF8(entry_hash) ) )
```

Note the signature covers the **hex string** of `entry_hash` (its UTF-8 bytes),
not the raw digest.

### 8.1 Verification

An Auditor verifies a chain by walking it in `seq` order. For each receipt it
**MUST** check, against the Authorizer's Ed25519 public key:

1. **Content** — `sha256(canonical_json(payload))` equals the stored `entry_hash`;
2. **Linkage** — `prev_hash` equals the previous receipt's `entry_hash`
   (`"GENESIS"` for the first);
3. **Signature** — `Ed25519_verify(pubkey, signature, UTF8(entry_hash))` succeeds.

The chain is valid iff every check passes for every receipt. Editing,
reordering, inserting, or deleting a receipt **from anywhere but the end** breaks
at least one check.

**Truncation is not detected by chaining alone (NORMATIVE caveat).** Removing the
most recent receipts leaves a shorter but internally consistent prefix that
verifies clean — the chain commits each receipt to its predecessor, not to the
*length* of the log. Likewise, an Auditor that only holds the Authorizer's public
key cannot detect this, and an attacker who holds the (local) signing key can
re-sign an arbitrary rewrite. The defense is to anchor the head externally:
persist the latest `(seq, entry_hash)` outside the log and reject a chain whose
last receipt doesn't match it. (Head-anchoring is specified normatively in
[spec.md §8.3 (v0.3)](../spec.md#83-head-anchoring-normative-03-draft--additive).)
High-assurance deployments **SHOULD** also keep the signing key in an HSM/KMS.

> **Hardened in 0.2.** 0.2 additionally requires a structurally invalid receipt
> (unparseable, or missing a signed field) to be treated as a verification
> *failure* at that position rather than an error that aborts the walk
> ([spec-v0.2.md §8.1](spec-v0.2.md#81-verification)).

Authoritative vectors: a valid chain ([`ctk/vectors/chain.jsonl`](../ctk/vectors/chain.jsonl),
expected [`chain.expected.json`](../ctk/vectors/chain.expected.json)) and a tampered
chain that MUST fail at `seq 0` ([`chain.tampered.jsonl`](../ctk/vectors/chain.tampered.jsonl)),
verifiable with [`ctk/vectors/signing_key.pub`](../ctk/vectors/signing_key.pub).

### 8.2 Schema versioning

The set of payload fields is part of the wire format. Any change to it is a
**breaking** change: it MUST bump this specification's version and the receipt
`schema` version together, or previously-signed chains stop verifying.

### 8.3 Head-anchoring

*Specified in v0.3.* The external head anchor — the required defense against
truncation and rollback per the §8.1 caveat — is defined normatively in
[spec.md §8.3](../spec.md#83-head-anchoring-normative-03-draft--additive).

## 10. Conformance

An implementation declares the highest protocol version (§2.1) it implements, and
**MUST** satisfy every clause at or below that version and reproduce that version's
CTK vectors.

**0.1**
- A conformant **Authorizer** MUST implement §3–§8 and reproduce the CTK
  `hashing` and `decisions` vectors, and produce chains that verify per §8.1.

## 11. Security considerations

- **'Firewall' is an analogy, not the model.** delego is an **action-authorization
  layer — a Policy Decision Point (PDP)** that renders `allow` / `needs_approval`
  / `deny` for proposed agent actions; it is **not** a network firewall. Where
  this document or related material reaches for "firewall", it is a loose analogy
  for "something that sits in front and gates", not a claim about packet
  filtering or network topology.
- **Confused deputy.** The protocol assumes the Agent can be redirected; the §7
  fingerprint guard ensures a redirected action cannot ride an approval issued
  for a different action. (0.1 does not defend against intent re-pointing or
  approval replay — see the §7 warning; closed in 0.2.)
- **No LLM in the decision path** (§2, §6) — an injection cannot argue its way
  past a deterministic evaluator.
- **Fail closed** — unmatched ⇒ `default` (`deny`); a matched-but-failed
  constraint ⇒ `deny`.
- **Query string** — in 0.1 the query is outside the fingerprint preimage
  (§4.2); decision-relevant values MUST travel in `params`. v0.3 specifies the
  Broker obligation and the eventual preimage fold.
- **Key management** — the Authorizer's Ed25519 signing key is the root of audit
  trust; its private half MUST NOT be exfiltrable. Compromise lets an attacker
  forge a chain. Truncation/rollback requires anchoring the head externally
  (§8.1; normative in v0.3 §8.3).
- **Concurrency** — a single logical chain has a single writer at a time;
  concurrent appenders can fork the chain. The reference v0.1 uses file-backed
  state and is **not** safe under concurrent writers (a single-writer daemon is
  planned).
